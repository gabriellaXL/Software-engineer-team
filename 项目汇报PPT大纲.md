## Slide 3 系统架构与部署方式

建议配图：前端、后端、数据库、文件/PDF 服务四层架构图。

页面内容：

- 前端：`frontend/index.html`、`frontend/app.js`、`frontend/styles.css` 单页应用
- 后端：Node.js + Express，按模块拆分路由与控制器
- 数据库：人大金仓 KingbaseES，使用 `pg` 驱动访问
- 认证：JWT + 角色权限中间件 + `token_version` 旧会话失效机制
- 文件与文档：DOCX 模板上传、附件保存、成绩单上传、Gotenberg/LibreOffice 转 PDF
- 部署：后端服务监听 3000 端口，并静态托管前端页面；Gotenberg 默认建议映射到 3001 端口

讲稿重点：

- 架构采用前后端职责分离，但部署上可以由 Express 同源托管前端，降低课程项目部署复杂度。
- 每个核心模块都有独立 API 路由，例如 `/api/auth`、`/api/policies`、`/api/process`、`/api/applications`、`/api/basic`、`/api/templates`、`/api/analysis`。

## Slide 7 重点闭环一：证明申请与审批

建议配图：学生申请 -> DOCX 填充/PDF 预览 -> 提交 -> 管理员审核 -> 学生查看结果。

页面内容：

流程：

1. 管理员在模板管理上传 DOCX 证明模板。
2. 学生在证明申请中选择模板，系统解析 `{姓名}`、`{学号}`、`{用途}` 等占位符。
3. 前端使用 docxtemplater/PizZip 填充模板，并调用后端 `/api/applications/convert-pdf` 转成 PDF 预览。
4. 学生提交申请，后端写入 `application` 表。
5. 管理员在审批管理查看申请详情，选择通过或退回，后端记录 `approval_record`。
6. 学生端申请记录更新，可查看审批意见和结果文件。

实现亮点：

- 关键占位符校验，避免生成姓名/学号不真实的证明。
- 申请记录和管理员审批列表使用同一后端数据源，形成闭环。
- 测试文档中相关缺陷已标记修复：申请创建、PDF 预览、提交后管理员可见、真实信息填充。

讲稿重点：

- 这是最能体现“从需求到可用系统”的主链路。
- 需要诚实说明：PDF 预览依赖 Gotenberg 或 LibreOffice，演示前要确认转换服务可用。

## Slide 8 重点闭环二：党团流程材料提交与审核

页面内容：

流程：

1. 管理员配置入党/入团流程节点，包括顺序、节点说明、开始/截止时间、提醒规则和附件。
2. 学生端展示当前可提交节点，只允许围绕当前阶段提交材料。
3. 学生可保存草稿或正式提交，支持附件上传。
4. 管理员端查看待审核流程材料，下载附件，填写审核意见。
5. 审核通过后更新学生进度；退回后学生端显示待补充并可重新提交。
6. 历史记录保留提交、审核和节点推进信息。

实现亮点：

- 入党/入团流程切换共用一套接口和状态逻辑。
- 后端限制只有当前节点、有效时间范围内才能正式提交。
- 审核结果会同步更新 `student_process_record` 和 `party_process_submission`。

讲稿重点：

- 这个模块解决“党团进度看不清、材料提交分散、审核留痕不足”的问题。
- 适合在答疑中解释事务处理、状态机、权限控制。

## Slide 9 重点闭环三：通知与精准推送

页面内容：

流程：

1. 管理员发布通知：标题、简介、正文、标签、目标年级、附件。
2. 发布前可以预览，也可以保存本地草稿。
3. 后端保存通知和目标年级范围。
4. 学生端根据当前学生年级过滤可见通知。
5. 学生查看详情、下载附件、标记已读或一键全读。
6. 管理员可删除通知，学生端同步不再展示。

实现亮点：

- 通知列表不是简单公告板，包含标签、目标年级、附件、已读状态。
- 草稿保存在浏览器本地，发布后写入后端数据库。
- 当前多渠道只实现站内通知，邮件/短信可作为后续扩展。

讲稿重点：

- 精准推送当前以“年级范围”为核心规则，已经能满足课程项目的基础定向需求。
- 后续可以扩展到专业、标签画像、重要通知短信提醒。

## Slide 10 重点闭环四：知识库与智能咨询

页面内容：

流程：

1. 管理员维护政策知识条目：标题、分类、关键词、标准答复、状态、附件。
2. 后端同步生成/更新问答数据，供学生端检索。
3. 学生端输入关键词或切换分类，前端请求 `/api/policies/search`。
4. 命中后展示知识条目、标准答复、维护部门、更新时间。
5. 若条目存在真实附件地址，则展示下载按钮；无附件时显示“暂无可下载附件”。

实现亮点：

- 把“智能咨询”落实为规则化知识库检索，避免大模型幻觉式回答。
- 附件下载逻辑已修复：只有存在真实地址时才显示真实下载入口。
- 管理员端支持附件上传到 `/uploads/policies/`。

讲稿重点：

- 这个功能的定位不是开放式聊天机器人，而是“政策知识库检索 + 标准答复 + 材料入口”。
- 适合回答老师关于准确性和可控性的提问。

## Slide 11 重点闭环五：成绩分析与培养方案

页面内容：

流程：

1. 管理员维护培养方案：年级、专业、状态、附件。
2. 学生在成绩分析页选择年级：当前支持 24 级、25 级。
3. 学生上传 CSV / Excel 成绩单。
4. 后端创建解析任务，异步解析课程、学分、模块。
5. 系统匹配培养方案或内置规则，计算学分达成度、缺失模块、必修课缺失、未分类课程。
6. 学生端展示分析摘要、缺失模块、选课建议和最近任务记录。

实现亮点：

- 后端异步处理成绩单上传，避免前端长时间等待。
- 可优先使用管理员维护的培养方案规则；没有方案时使用内置规则完成演示。
- 结果包含置信度、识别课程数、通过学分等可解释指标。

讲稿重点：

- 该模块是扩展功能，目前可以体验上传、解析任务和分析结果展示。
- 根据测试文档，解析准确性仍依赖文件格式和培养方案规则，可作为后续优化点。

## Slide 12 设计理念：为什么这样设计

页面内容：

总体设计思想：

- 双端协同：学生端办理，管理员端配置和审核。
- API 模块化：认证、知识库、流程、申请、通知、模板、成绩分析分别拆分。
- 数据驱动界面：通知、知识库、流程节点、模板、培养方案都由后端数据驱动。
- 可演示优先：课程项目有限周期内优先打通核心业务闭环，再逐步增强智能化。

用户体验设计：

- 学生端首页聚合待办、未读通知、流程进度和快捷入口。
- 管理员端采用工作台风格，表格 + 右侧表单便于重复操作。
- 关键操作都有 Toast 提示和空状态提示。
- 移动端学生导航有底部菜单，适配手机浏览。

关键设计决策：

- 使用 JWT 做登录态，配合 `token_version` 解决改密码后旧会话仍可用的问题。
- 证明模板采用 DOCX 占位符，而不是固定写死 PDF，提高模板复用能力。
- 成绩分析先支持 CSV/Excel 表格，PDF 成绩单转换作为后续扩展，降低解析风险。

讲稿重点：

- 我们没有一开始追求“全自动智能化”，而是先保证数据可维护、流程可追踪、审批可留痕。

## 7 分钟答疑准备

可能问题 1：为什么没有真正用大模型做智能问答？

回答要点：

- 当前系统优先保证政策回答准确可控。
- 政策类问答不适合直接由大模型自由生成，否则可能产生幻觉。
- 所以采用“管理员维护知识库 + 学生检索标准答复”的方案，后续可在知识库召回基础上接入 RAG。

可能问题 2：证明 PDF 生成失败怎么办？

回答要点：

- PDF 转换依赖 Gotenberg 或 LibreOffice。
- 前端先做模板占位符校验，减少错误模板进入转换流程。
- 后端先调用 Gotenberg，失败后尝试 LibreOffice fallback。
- 演示前会准备固定模板和运行好的转换服务。

可能问题 3：如何保证权限安全？

回答要点：

- 所有受保护 API 都经过 JWT 认证。
- 不同角色通过 `authorize` 中间件限制接口访问。
- 修改密码后 `token_version` 自增，旧 Token 在下一次请求时被拒绝。
- 当前仍有密码明文存储问题，后续应改为哈希存储。

可能问题 4：哪些功能没有完成？

回答要点：

- 理论自测、奖励荣誉展示没有独立实现。
- 多渠道通知目前是站内消息，未接入邮件/短信。
- 数据导入导出后端能力存在，但前端入口主要接入知识库导入。
- 成绩分析能完成上传和规则比对，但准确率依赖文件格式和培养方案维护。

可能问题 5：测试是否充分？

回答要点：

- 已基于互测反馈整理 9 个缺陷，并全部标记修复。
- 测试覆盖登录、证明申请、PDF 预览、知识库附件、用户校验、旧会话失效等关键点。
- 目前主要是手工测试和回归记录，后续需要补充自动化测试、接口测试集合和正式环境最终回归。

可能问题 6：为什么不用 Vue/React？

回答要点：

- 课程项目周期有限，原生单页应用能降低构建和部署复杂度。
- 当前功能重点是业务闭环和接口联调。
- 如果后续继续迭代，建议迁移到 Vue/React 并拆分组件。

可能问题 7：系统最大亮点是什么？

回答要点：

- 不是单个页面，而是多个真实业务闭环：知识库维护到学生咨询、通知发布到精准可见、证明申请到审批留痕、党团流程材料到审核推进、培养方案到成绩分析。
- 同时有完整软件工程产物：需求、设计、代码、测试、手册、部署说明。
