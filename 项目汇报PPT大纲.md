## 1. 项目概览：学院学生事务的一站式服务平台

**建议用时：40 秒**

**页面要点**

- 项目名称：学院学生综合服务与党团管理平台
- 面向对象：学生、班团骨干、管理老师、学院领导等多角色用户
- 核心定位：把政策咨询、党团流程、通知公告、证明申请、成绩分析等学生事务集中到一个 Web 平台
- 当前交付形态：学生端 + 管理员端 + 后端 REST API + Kingbase 数据库

**讲解重点**

- 这个项目不是单一功能系统，而是围绕学院学生工作场景做的综合服务平台。
- 学生端关注“查、办、看结果”；管理员端关注“维护、配置、审核、批量处理”。
- 后端统一提供 API，并由数据库保存用户、流程、审批、通知、模板等核心数据。

## 2. 项目背景与需求痛点

**建议用时：45 秒**

**页面要点**

- 传统学生事务依赖人工沟通、微信群通知和线下材料流转
- 学生常见问题重复咨询，政策材料分散，办理流程不透明
- 管理老师需要频繁维护通知、审批申请、跟踪党团流程
- 需求目标：电子化办理、知识库咨询、精准通知、流程可视化、数据可追踪

**讲解重点**

- 需求文档中明确了几个典型痛点：信息分散、重复劳动多、学生不清楚流程、材料审批成本高。
- 我们将需求拆成学生侧和管理侧两条线：学生侧提升办理效率，管理侧降低维护和审核成本。
- 项目预计服务学院约 1200 到 1300 名学生，并考虑约 300 人同时在线的访问规模。

- 左侧：微信群、人工问答、线下材料；右侧：统一平台、知识库、在线审批

## 3. 总体功能结构

**建议用时：55 秒**

**页面要点**

- 学生端功能：
  - 首页概览、智能咨询、党团流程、通知公告、证明申请、成绩分析、模板下载、个人信息维护
- 管理员端功能：
  - 后台首页、用户管理、知识库管理、通知管理、流程配置、审批管理、模板管理、培养方案管理
- 公共能力：
  - 登录注册、JWT 鉴权、角色权限控制、文件上传、Excel 导入导出、模板与证明文件处理

**讲解重点**

- 学生端以任务为中心：进入首页后可以直接看到待办、未读通知、流程进度和证明记录。
- 管理端以后台工作台为中心：可以管理用户、维护知识库、发布通知、配置流程、审批材料。
- 两端共享同一套后端数据，避免学生端和管理员端状态不一致。

**建议配图**

- 功能矩阵：学生端 / 管理员端 / 公共能力三列

## 4. 技术架构与运行方式

**建议用时：60 秒**

**页面要点**

- 前端：HTML + CSS + JavaScript 单页应用
- 后端：Node.js + Express
- 数据库：人大金仓 KingbaseES，使用 `pg` 驱动连接
- 接口：RESTful API
- 认证：JWT Token + 角色权限中间件
- 文件与文档能力：
  - `multer` 处理上传
  - `xlsx` 支持 Excel 导入导出
  - `docxtemplater` / `docx-preview` / `html2pdf` 支持模板填充和预览
  - Gotenberg 或 LibreOffice 用于 DOCX 转 PDF

**讲解重点**

- 当前采用前后端分离思想，但部署时后端也托管前端静态页面，因此课程服务器上只需要启动一个后端服务即可访问页面和 API。
- 后端按模块拆分为 routes、controllers、middleware、config、bootstrap，便于维护。
- Kingbase 兼容 PostgreSQL 协议，因此项目通过 `pg` 驱动连接数据库。

**建议配图**

- 架构图：
  - 浏览器学生端 / 管理端
  - Express API
  - JWT 鉴权
  - 业务模块
  - Kingbase 数据库
  - 文件存储 / Gotenberg

## 6. 学生端核心闭环

**建议用时：70 秒**

**页面要点**

- 首页：待办提醒、未读通知、流程进度、证明记录
- 智能咨询：关键词检索政策条目和标准问答
- 党团流程：切换入党/入团流程，查看时间线，提交材料，保存草稿，查看历史
- 通知公告：按标签筛选，查看详情，标记已读
- 证明申请：选择 DOCX 模板，填写用途，预览 PDF，提交审批，查看记录
- 成绩分析：上传成绩单，和培养方案学分要求比对，输出缺失模块与建议

**讲解重点**

- 学生端的设计目标是减少“问老师、找文件、线下跑”的成本。
- 以证明申请为例：学生先维护个人信息，再选择模板，系统读取模板占位符并生成填写项，预览后提交，管理员审核后状态回到学生端。
- 以党团流程为例：学生只能对当前激活节点提交材料，系统会根据节点开始时间、截止时间和审核结果控制流转。

## 7. 管理员端核心闭环

**建议用时：70 秒**

**页面要点**

- 后台首页：待审批事项、模板数量、知识库条目、学生数量
- 用户管理：新增、编辑、删除用户，维护角色和状态
- 知识库管理：新增/编辑政策条目，维护关键词、分类、附件
- 通知管理：标签、年级、附件、草稿、发布和批量删除
- 流程配置：维护入党/入团节点、开始/截止时间、提醒规则、节点附件
- 审批管理：审核证明申请和党团流程材料，支持通过或退回补充
- 模板管理：上传 `.docx` 模板，供学生端证明申请使用
- 培养方案：维护方案文件和学分比对依据

**讲解重点**

- 管理员端是学生端功能的支撑后台，学生端看到的数据大多由管理员配置或审核。
- 流程配置体现了项目的可扩展性：流程节点不是写死在前端，而是保存在数据库中。
- 通知管理体现精准推送：通知可以带标签和目标年级，学生端只展示自己可见的通知。

## 8. 三条重点业务流程演示

**建议用时：80 秒**

**页面要点**

**流程一：登录与身份识别**

- 用户输入账号密码
- 后端校验 `tb_user`
- 生成 JWT
- 前端携带 Token 调用后续接口

**流程二：学生提交申请 -> 管理员审批**

- 学生选择模板并提交申请
- 后端写入 `application`
- 管理员在审批管理页查看
- 审批结果写入 `approval_record`
- 学生端申请状态更新为已通过或待补充

**流程三：党团流程材料提交 -> 审核 -> 进度更新**

- 管理员配置流程节点
- 学生在当前节点提交材料
- 管理员审核通过或退回
- 通过后更新 `student_process_record`，学生端时间线同步变化

**讲解重点**

- 这三条链路最适合现场演示，因为它们能覆盖认证、权限、数据库读写、前后端联动和状态流转。
- 其中党团流程和证明申请都体现了“申请人提交、管理端审核、结果回到学生端”的闭环。

**建议配图**

- 画一个横向流程图：学生端 -> API -> 数据库 -> 管理端 -> 数据库 -> 学生端

## 9. 实现亮点与技术难点

**建议用时：60 秒**

**页面要点**

- 数据驱动流程：
  - 入党/入团流程节点可由管理员配置
  - 支持节点时间、提醒规则、附件、材料审核
- 权限控制：
  - JWT 鉴权
  - `student`、`student_leader`、`admin`、`teacher`、`leader` 多角色控制
- 文件处理：
  - DOCX 模板上传、下载、填充、预览
  - Gotenberg 转 PDF，失败后可尝试 LibreOffice 兜底
- 批量数据处理：
  - 用户、流程节点、通知、政策、培养方案支持 Excel 模板导入导出
- 异步成绩分析：
  - 上传成绩单后创建任务
  - 解析课程和学分
  - 与培养方案模块要求比对

**讲解重点**

- 最大的技术难点不是某一个算法，而是多个业务模块的状态一致性。
- 我们用数据库作为统一状态源，并通过 REST API 将学生端和管理员端连接起来。
- 文件和模板处理是项目中比较有展示价值的部分，体现了从传统 Word/PDF 材料到线上办理的迁移。

**建议配图**

**建议用时：35 秒**

  - 密码目前为课程演示用明文比对，后续应改为哈希存储

# 备用页：7 分钟答辩提问准备

## Q1：为什么没有直接接入学校现有系统？

**回答要点**

- 需求阶段已确认当前无法直接对接校级“微人大”等系统接口。
- 因此本项目采用导入导出方式作为过渡方案，支持 Excel/Word/PDF 数据维护。
- 架构上保留 REST API 和模块边界，未来如果有接口权限，可以替换数据来源而不推翻前端功能。

## Q2：智能咨询是不是用了大模型？

**回答要点**

- 当前版本以政策知识库和标准问答检索为主，不依赖大模型自由生成。
- 这样可以降低政策答复“幻觉”风险，适合学生事务这种要求准确性的场景。
- 后续可以引入 AI 做语义匹配或纠错，但最终答复仍应绑定标准政策条目。

## Q3：权限是怎么控制的？

**回答要点**

- 登录成功后后端签发 JWT，前端后续请求携带 `Authorization: Bearer <token>`。
- 后端中间件先验证 Token，再根据角色数组控制接口访问。
- 例如学生可以提交申请，管理员/教师可以审核申请，领导或教师可以管理通知和用户。

## Q4：党团流程为什么说是可配置的？

**回答要点**

- 流程节点保存在 `party_process_node` 表中，而不是写死在前端。
- 管理员可以维护节点名称、顺序、流程类型、开始时间、截止时间、提醒规则和附件。
- 学生端根据数据库返回的节点和个人进度生成时间线。

## Q5：证明申请和电子证明是如何实现的？

**回答要点**

- 管理员先上传 `.docx` 模板。
- 学生选择模板并填写用途和动态字段，前端可预览填充后的 PDF。
- 提交后进入 `application` 申请表，管理员审核后写入 `approval_record`。
- DOCX 转 PDF 使用 Gotenberg，失败时可尝试 LibreOffice 兜底。

## Q6：成绩分析的核心逻辑是什么？

**回答要点**

- 学生上传成绩单后，系统创建 `transcript_task` 异步任务。
- 解析课程、学分和模块信息后，与 `course_requirement` 中的培养方案要求比对。
- 最终生成 `analysis_result`，包括缺失模块、预警等级和选课建议。

## Q7：当前系统还有哪些不足？

**回答要点**

- 课程演示版本仍需补强安全和工程化能力，例如密码哈希、统一错误结构、自动化测试、操作审计。
- PDF/Excel 等复杂文件解析可以继续增强。
- 生产环境还需要接入统一身份认证、HTTPS、日志监控和更严格的数据权限。
