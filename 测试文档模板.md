## 1 测试概述

### 1.1 编写目的

本文档用于记录“学院学生综合服务与党团管理平台”在课程互测阶段由其他小组反馈出的缺陷情况，并结合项目需求文档、用户使用说明文档、管理员端/学生端基本功能说明以及当前代码实现，对缺陷进行复现过程整理、影响分析和改进建议归纳。

本文档的重点不是重新罗列全部功能，而是围绕互测阶段已经暴露的问题，说明这些问题对应的测试依据、测试步骤、实际结果、期望结果和修复状态，为后续缺陷修复、回归测试、最终验收和文档转换提交提供依据。

### 1.2 测试范围

本次测试范围覆盖互测反馈中涉及的核心业务模块，包括：

* 用户登录、退出与密码修改后的会话控制。
* 管理员端用户管理，包括新建用户、必填字段校验和手机号异常输入校验。
* 学生端智能咨询与知识库附件下载。
* 学生端证明申请，包括新建申请、PDF 预览、提交申请、申请记录展示和管理员端审批可见性。
* 学生端党团流程材料操作中的按钮反馈与提示信息一致性。

重点验证内容包括：

* 功能正确性：功能入口、表单提交、PDF 预览、申请记录等是否符合使用说明和需求预期。
* 异常处理能力：空姓名、异常手机号、缺失附件、PDF 转换失败等情况是否有明确拦截或提示。
* 用户操作流程完整性：学生提交申请后，学生端和管理员端是否能形成闭环。
* 安全性：密码修改后旧会话是否失效，旧页面是否仍可执行受保护操作。
* 文档与系统一致性：页面行为是否与用户使用说明、基本功能文档和需求文档描述一致。

## 2 测试设计

### 2.1 测试策略

本次测试以黑盒功能测试为主，结合部分代码阅读进行缺陷原因分析。测试执行依据主要包括：

* 《软件需求文档》中对用户登录、知识库、证明自动生成、审批流、异常处理和安全约束的描述。
* 《用户使用说明文档模板.md》中对学生端和管理员端操作流程的说明。
* `使用手册/学院学生综合服务与党团管理平台基本功能文档与使用说明.md` 中对学生端功能、测试账号和已知限制的说明。
* `使用手册/管理员端基本功能文档与使用说明.md` 中对管理员端用户管理、知识库、审批管理、模板管理等功能的说明。
* 当前代码中与认证、用户管理、知识库附件、证明申请和模板 PDF 预览相关的前后端实现。

测试方法包括：

* 正常流程测试：验证证明申请新建、预览、提交、记录展示、管理员审批入口等主流程。
* 异常流程测试：验证空姓名、异常手机号、附件下载失败、PDF 转换失败等异常情况。
* 边界值测试：针对手机号输入 `0.5`、管理员新建用户姓名为空等边界/非法输入进行测试。
* 安全性测试：通过双页面登录场景验证密码修改后旧会话是否被强制失效。
* 回归测试：对第1阶段已修复的缺陷记录修复状态，后续需再次执行同类步骤确认修复未回退。

测试优先级按影响范围划分：

* 高优先级：影响账号安全、核心申请闭环、证明 PDF 正确性的缺陷。
* 中优先级：影响数据校验、附件下载、管理员创建用户准确性的缺陷。
* 一般优先级：页面提示与按钮含义不一致等用户体验和一致性缺陷。

### 3.2 缺陷分类分析

从缺陷分布看，问题主要集中在以下几类：

* **核心业务闭环缺陷**：证明申请从“选择模板、预览 PDF、提交申请、学生端展示记录、管理员端可审批”应形成完整闭环。第1阶段多个缺陷说明该链路早期存在不稳定问题，虽然已标记修复，但仍应作为回归测试重点。
* **数据填充与模板处理缺陷**：证明申请 PDF 中学生姓名、学号等信息显示为 `××`，说明模板占位符替换或学生档案读取存在不一致。当前版本已在预览前增加关键占位符检测，避免使用缺少 `{姓名}`、`{学号}` 的异常模板继续生成误导性 PDF。
* **输入校验缺陷**：手机号允许输入 `0.5`、管理员新建用户姓名为空仍成功，说明前后端校验不完整。当前版本已补充管理员端新建用户姓名必填校验，并取消姓名为空时用账号兜底的隐式行为。
* **附件下载链路缺陷**：智能咨询知识库中显示附件下载入口，但点击失败。当前版本已调整前端展示逻辑，对无真实附件地址的知识库条目不再展示可点击下载按钮，而是明确显示暂无可下载附件；存在有效附件地址时仍保留下载入口。
* **会话安全缺陷**：修改密码后仅当前页面退出，另一旧页面仍可用旧 Token 访问接口。当前版本已增加 Token 版本校验机制，修改密码后旧 Token 会在后续请求中被服务端拒绝，前端收到失效响应后清理登录状态并跳转登录。
* **提示文案一致性缺陷**：保存草稿按钮提示“当前不可提交”，会使用户误解操作结果。此类问题虽然不一定导致数据损坏，但会降低可用性并影响互测评价。

2. 打开两个页面，使两个页面同时处于登录状态。
3. 在页面 A 点击左侧功能栏最下方的“修改个人信息”，进入个人信息页面。
4. 在页面 A 底部“账号安全”区域修改密码，测试时新密码为 `1234567`。
5. 页面 A 刷新并提示需要重新登录。
6. 页面 B 仍显示为登录后的正常页面。
7. 在页面 B 进行操作，例如进入成绩单解析、上传文件、解析，或点击、新建申请等。
8. 测试后已将密码重置回 `123456`。

**实际结果**：

密码修改后，页面 A 正常退出登录；页面 B 仍可继续执行旧会话下的操作，未提示密码已修改或要求重新登录。

**期望结果**：

修改密码后，所有基于旧密码认证的会话应立即失效。任意旧页面再次请求受保护接口时，应返回未认证或要求重新登录。

**影响范围**：

该缺陷属于安全性问题。若用户在公共设备或多个页面中登录，修改密码后旧会话仍有效，会降低密码修改的安全意义。

**原因分析**：

当前前端使用 SessionStorage 保存 Token；后端使用 JWT 验证 Token。若 JWT 中只包含用户 ID、角色等静态信息，且服务端不保存 Token 版本或密码版本，则密码更新后已签发 Token 在过期前仍可通过校验。页面 A 主动清除自身 SessionStorage 只能影响当前标签页，不能让服务端拒绝页面 B 的旧 Token。

**修复措施**：

1. 在用户表中增加 `token_version` 字段，用于记录用户当前有效 Token 版本。
2. 用户登录时将当前 `token_version` 写入 JWT。
3. 认证中间件在每次受保护请求中查询用户当前 `token_version`，若 Token 中版本与数据库版本不一致，则拒绝请求并返回登录失效提示。
4. 修改密码接口成功后递增该用户 `token_version`，使修改密码前签发的旧 Token 全部失效。
5. 前端统一处理 401/Token 失效响应，清理本页面登录状态并跳转登录页，避免继续显示“未知用户”等异常状态。

**回归测试结果**：

本地双页面测试中，页面 A 修改密码后，页面 B 使用旧 Token 再次刷新或发起受保护请求时会被判定为登录失效，并跳转登录页。旧页面不再能够继续执行有效后端操作，也不再出现跳转到未知同学界面的异常。若页面 B 完全不刷新且不发请求，原有静态界面可能短暂保留；一旦发生接口请求，即会被强制重新认证。

**证据材料**：

## 4 测试结论与建议

### 4.1 测试结论

根据互测反馈，本系统已覆盖学生端和管理员端的主要功能入口，证明申请、智能咨询、用户管理、党团流程、密码修改等核心模块均已进入可测试状态。第1阶段发现的 5 个缺陷均已标记为已修复，项目组已对早期阻断性问题进行了处理。

第2阶段发现的 4 个缺陷分别涉及知识库附件下载、证明 PDF 数据填充、密码修改后的旧会话失效、管理员新建用户必填校验。当前版本已完成对应修复：智能咨询无附件条目不再展示无效下载入口；证明申请预览前增加关键占位符检测；密码修改后旧 Token 会在后续请求中失效；管理员新建用户时姓名字段改为必填并增加后端校验。

综合判断：当前系统可基本支持主要功能演示，互测阶段反馈的 9 个缺陷均已完成修复标记。考虑到智能咨询附件、证明模板和部分学生信息依赖线上数据库及上传文件，建议结论为：**基本通过测试，已完成缺陷修复，需在正式部署环境中完成最终回归验证后进入最终提交版本。**

### 4.2 改进建议

1. **完善后端强校验**：所有关键字段校验不能只依赖前端。管理员新建用户、个人信息维护、手机号、姓名、学号、密码等字段都应在后端接口统一校验。

2. **强化证明模板规范**：模板管理应要求管理员上传包含标准花括号占位符的 `.docx` 文件，并在上传时检测 `{姓名}`、`{学号}` 等关键字段。对不符合规范的模板应提示重新上传。

3. **稳定 PDF 生成链路**：证明申请的 DOCX 填充、PDF 转换、预览展示、提交保存和管理员审批应作为一条完整链路进行回归测试。建议保留一份固定测试模板和固定学生账号作为回归基准。

4. **保持知识库附件下载逻辑一致**：知识库条目如果展示下载按钮，必须保证附件地址真实可访问；无附件时应隐藏按钮或明确提示暂无附件。演示入口和真实下载入口应在文案上区分。

5. **持续验证密码修改后的旧会话失效机制**：服务端已增加 Token 版本校验，使修改密码后旧 JWT 在后续请求中失效；前端已统一处理 401 响应并跳转登录页。后续应在正式部署环境中继续验证多标签页、多浏览器和不同账号场景。

6. **补充回归测试清单**：对第1阶段已修复缺陷仍需在最终版本中再次执行，防止证明申请、PDF 预览、申请记录等核心流程回退。

7. **补充截图证据**：互测记录中已有图片证据，建议后续转换 Word 前将关键截图补充到 3.3 对应缺陷小节下，尤其是第2阶段 4 个缺陷的修复前后截图。

8. **保持文档与系统一致**：若某些功能当前仅为演示交互，应在用户手册和测试文档中明确说明；若页面展示真实按钮，则应保证其真实可用，避免互测时被判定为逻辑缺陷或文档不一致。
